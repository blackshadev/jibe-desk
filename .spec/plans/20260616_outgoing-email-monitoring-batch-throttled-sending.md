# Plan: Outgoing Email Monitoring, Batch & Throttled Sending

## Overview

Implement a comprehensive outgoing email tracking and monitoring system. All outgoing emails are logged to an `outgoing_emails` table and viewable in a dedicated `OutgoingEmailResource` Filament resource and on the `MemberResource` via a relation manager. Invoice batch emails are sent in background via queued job batches with rate limiting. Existing registration emails are converted to queued background sending. Emails not addressed to a member are classified as "system" emails and filterable as such.

**Strategy**: Event-driven mail tracking using Laravel's native `MessageSending`/`MessageSent` events, supplemented by custom message headers for mailable metadata. This aligns with the existing DDD event/listener architecture discovered via `bootstrap/app.php`.

**Key design decisions:**
- `tracking_id` is a UUID generated by `LogOutgoingMail` and injected as an `X-Outgoing-Email-Id` header into the Symfony message. This guarantees a 1:1 mapping between the DB record and the sent message, even when the same mailable is sent multiple times to the same recipient.
- `member_id` FK enables direct relationship with the `Member` model for the relation manager
- `mailable_class` stores the FQCN of the mailable for identification; mail without this header (e.g. Fortify) has it as null
- `status` stored as string, cast to `OutgoingEmailStatus` enum on the model
- `batch_id` links to `job_batches.id` for batch progress tracking
- Polymorphic `related_model_type`/`related_model_id` for linking to Invoice, etc.
- Emails without a `member_id` are classified as "system" emails

## 2. Create the `OutgoingEmailTrackingId` Value Object

### File: `app/Domain/Mail/OutgoingEmailTrackingId.php` (new)

A UUID-based value object that uniquely identifies a single outgoing email record. Generated in `LogOutgoingMail`, stored on the DB record, and injected as an `X-Outgoing-Email-Id` header into the Symfony message. `MarkMailAsSent` reads this header back for a precise 1:1 lookup.

```php
<?php

declare(strict_types=1);

namespace App\Domain\Mail;

use Illuminate\Support\Str;
use Override;
use Stringable;
use Webmozart\Assert\Assert;

final readonly class OutgoingEmailTrackingId implements Stringable
{
    public function __construct(
        public string $value,
    ) {
        Assert::uuid($value);
    }

    public static function generate(): self
    {
        return new self((string) Str::uuid());
    }

    #[Override]
    public function __toString(): string
    {
        return $this->value;
    }
}
```

**Pattern notes:**
- Follows the `InvoiceNumber` VO pattern: `final readonly class`, public `string $value`, validated, `Stringable`
- `Assert::uuid()` validates the format (same library used by `NumericId` and `InvoiceNumber`)
- `generate()` static factory creates a fresh UUID — the primary construction path
- `__toString()` allows direct use in string context (e.g., header values, DB storage)

### File: `app/Infrastructure/Mail/OutgoingEmailRepositoryDb.php` (new)

```php
<?php

declare(strict_types=1);

namespace App\Infrastructure\Mail;

use App\Domain\Mail\NewOutgoingEmail;
use App\Domain\Mail\OutgoingEmailRepository;
use App\Domain\Mail\OutgoingEmailStatus;
use App\Models\Member;
use App\Models\OutgoingEmail;
use Override;

final class OutgoingEmailRepositoryDb implements OutgoingEmailRepository
{
    #[Override]
    public function create(NewOutgoingEmail $email): void
    {
        OutgoingEmail::query()->create([
            'tracking_id' => $email->trackingId,
            'mailable_class' => $email->mailableClass,
            'recipient_email' => $email->recipientEmail,
            'recipient_name' => $email->recipientName,
            'subject' => $email->subject,
            'status' => OutgoingEmailStatus::Queued,
            'member_id' => $email->memberId,
            'queued_at' => $email->queuedAt,
        ]);
    }

    #[Override]
    public function markLatestAsSent(TrackingId $trackingId, string $messageId): void
    {
        OutgoingEmail::query()
            ->where('tracking_id', $trackingId->value)
            ->where('status', OutgoingEmailStatus::Queued)
            ->latest('queued_at')
            ->first()
            ?->update([
                'status' => OutgoingEmailStatus::Sent,
                'message_id' => $messageId,
                'sent_at' => now(),
            ]);
    }

    #[Override]
    public function findMemberIdByEmail(string $email): ?MemberId
    {
        $id = Member::query()
            ->where('email', $email)
            ->value('id');
            
        return $id !== null ? new MemberId($id) : null;
    }
}
```

## 8. Create Mail Tracking Listeners

These listeners react to Laravel's native mail events. They live in `app/Domain/Mail/Listeners/` which is within the `app/Domain` event discovery path configured in `bootstrap/app.php`. They depend only on the `OutgoingEmailRepository` interface — no direct Eloquent usage.

### File: `app/Domain/Mail/Listeners/LogOutgoingMail.php` (new)

Listens to `Illuminate\Mail\Events\MessageSending`. Creates an `OutgoingEmail` record with status `queued` via the repository.

```php
<?php

declare(strict_types=1);

namespace App\Domain\Mail\Listeners;

use App\Domain\Mail\NewOutgoingEmail;
use App\Domain\Mail\OutgoingEmailRepository;
use Illuminate\Mail\Events\MessageSending;

final readonly class LogOutgoingMail
{
    public function __construct(
        private OutgoingEmailRepository $outgoingEmailRepository,
    ) {}

    public function handle(MessageSending $event): void
    {
        $message = $event->message;

        $recipientEmail = $this->extractRecipientEmail($message);
        $mailableClass = $this->extractHeader($message, 'X-Mailable-Class');
        $trackingId = $this->extractHeader($message, 'X-Outgoing-Email-Id');

        if ($recipientEmail === null || $trackingId === null || $mailableClass === null) {
            return;
        }

        $memberId = $this->outgoingEmailRepository->findMemberIdByEmail($recipientEmail);

        $this->outgoingEmailRepository->queue(new NewOutgoingEmail(
            trackingId: new TrackingId($trackingId),
            mailableClass: $mailableClass,
            recipientEmail: $recipientEmail,
            recipientName: $this->extractRecipientName($message),
            subject: $message->getSubject() ?? '',
            memberId: $memberId,
            queuedAt: now(),
        ));
    }

    private function extractRecipientEmail(\Symfony\Component\Mime\Email $message): ?string
    {
        $to = $message->getTo();

        return $to !== [] ? $to[0]->getAddress() : null;
    }

    private function extractRecipientName(\Symfony\Component\Mime\Email $message): ?string
    {
        $to = $message->getTo();

        if ($to === []) {
            return null;
        }

        $name = $to[0]->getName();

        return $name !== '' ? $name : null;
    }

    private function extractHeader(\Symfony\Component\Mime\Email $message, string $name): ?string
    {
        $header = $message->getHeaders()->get($name);

        return $header?->getBodyAsString();
    }
}
```

### File: `app/Domain/Mail/Listeners/MarkMailAsSent.php` (new)

Listens to `Illuminate\Mail\Events\MessageSent`. Updates the latest queued `OutgoingEmail` record to `sent` via the repository.

```php
<?php

declare(strict_types=1);

namespace App\Domain\Mail\Listeners;

use App\Domain\Mail\OutgoingEmailRepository;
use Illuminate\Mail\Events\MessageSent;

final readonly class MarkMailAsSent
{
    public function __construct(
        private OutgoingEmailRepository $outgoingEmailRepository,
    ) {}

    public function handle(MessageSent $event): void
    {
        $messageId = $event->sent->getMessageId();
        $trackingId = $this->extractHeader($message, 'X-Outgoing-Email-Id');
        
        if ($trackingId === null) {
            return;
        }

        $this->outgoingEmailRepository->markLatestAsSent(new TrackingId($trackingId), $messageId);
    }

    private function extractHeader(\Symfony\Component\Mime\Email $message, string $name): ?string
    {
        $header = $message->getHeaders()->get($name);

        return $header?->getBodyAsString();
    }
}
```

**Important notes:**
- `MessageSending`/`MessageSent` fire when the mail is actually being **sent** by the queue worker, not when queued. This is correct — the `queued` status is set right before the SMTP transaction, and `sent` right after.
- For mailables without the `X-Mailable-Class` header (e.g., Fortify password reset), `mailable_class` will be null.
- Both listeners receive `OutgoingEmailRepository` via constructor injection — Laravel's container resolves the `#[Autowire]`-tagged interface to `OutgoingEmailRepositoryDb`.
- Member matching is done through the repository's `findMemberIdByEmail()` method.

## 10. Create the `InvoiceEmailLine` Value Object

### File: `app/Domain/Invoices/InvoiceEmailLine.php` (new)

A domain value object representing a single invoice line's display data for email rendering. Follows the `SepaExportInvoice` VO pattern: `final readonly class` with typed constructor properties and a derived formatting method.

```php
<?php

declare(strict_types=1);

namespace App\Domain\Invoices;

final readonly class InvoiceEmailLine
{
    public function __construct(
        public string $description,
        public float $quantity,
        public CompoundPrice $price,
        public CompoundPrice $subtotal,
    ) {}

    /** Quantity formatted without trailing zeros (e.g. "1", "2.5"). */
    public function formattedQuantity(): string
    {
        return rtrim(rtrim(number_format($this->quantity, 2, ',', '.'), '0'), ',');
    }
}
```

**Pattern notes:**
- `final readonly class` — same as `SepaExportInvoice`, `CompoundPrice`, `InvoiceNumber`
- Uses `CompoundPrice` for `price` and `subtotal` — leverages the existing `__toString()` method which calls `PriceFormatter::format()`, so `{{ $line->price }}` in Blade renders `"€ 12,50"`
- `formattedQuantity()` — strips trailing zeros from the decimal quantity (e.g., `1.00` → `"1"`, `2.50` → `"2,5"`)
- No Eloquent or framework dependencies — pure domain VO

```php
        public readonly int $invoiceId,
        public readonly string $recipientEmail,
    ) {
        $this->onQueue('invoices');
    }

    /** @return array<int, object> */
    public function middleware(): array
    {
        return [new RateLimited('invoice-emails')];
    }

    public function handle(): void
    {
        if ($this->batch()?->cancelled()) {
            return;
        }

          = Invoice::with(['lines', 'member', 'invoiceBatch'])->findOrFail($this->invoiceId);

        $lines = $invoice->lines->map(static fn (InvoiceLine $line) => new InvoiceEmailLine(
            description: $line->description,
            quantity: (float) $line->quantity,
            price: $line->compoundPrice,
            subtotal: $line->subTotal,
        ))->all();

        $mail = new InvoiceMail(
            invoiceId: InvoiceId::create($invoice->id),
            invoiceNumber: $invoice->invoice_number,
            memberName: $invoice->recipient_name,
            memberEmail: $this->recipientEmail,
            recipientAddress: $invoice->recipient_address,
            total: $invoice->total,
            invoiceDate: $invoice->date->format('d-m-Y'),
            sepaTransferDate: $invoice->invoiceBatch->invoice_date->format('d-m-Y'),
            lines: $lines,
        );

        Mail::to($this->recipientEmail)
            ->send($mail);
    }
}
```

**Key design decisions:**
- Uses `Batchable` trait for job batch integration (progress tracking, cancellation)
- `RateLimited('invoice-emails')` middleware — throttled to 20/min via rate limiter
- Queue name `invoices` — dedicated queue for invoice emails
- `tries = 3`, `maxExceptions = 2` — retry on transient failures
- Checks `$this->batch()?->cancelled()` before sending — supports batch cancellation
- Creates `InvoiceEmailLine` VOs instead of associative arrays — type-safe, domain-typed line data
- Uses `$line->compoundPrice` and `$line->subTotal` (Eloquent computed attributes returning `CompoundPrice`) — no manual `PriceFormatter` calls needed; formatting is handled by `CompoundPrice::__toString()` in the Blade template

```php
<?php

declare(strict_types=1);

namespace App\Infrastructure\Invoices;

use App\Domain\Invoices\InvoiceBatchEmailDispatcher;
use App\Domain\Invoices\InvoiceBatchId;
use App\Domain\Invoices\InvoiceBatchRepository;
use App\Domain\Invoices\InvoiceEmailRecipient;
use App\Jobs\SendInvoiceEmail;
use Illuminate\Support\Facades\Bus;
use Override;

final readonly class InvoiceBatchEmailDispatcherImpl implements InvoiceBatchEmailDispatcher
{
    public function __construct(
        private InvoiceBatchRepository $batchRepository,
    ) {}

    #[Override]
    public function dispatch(InvoiceBatchId $batchId): void
    {
        $recipients = $this->batchRepository->getInvoiceEmailRecipients($batchId);

        if ($recipients === []) {
            return;
        }

        $jobs = array_map(
            static fn (InvoiceEmailRecipient $recipient) => new SendInvoiceEmail(
                invoiceId: $recipient->invoiceId->value,
                recipientEmail: $recipient->recipientEmail,
            ),
            $recipients,
        );

        Bus::batch($jobs)
            ->name('invoice-batch-' . $batchId->value)
            ->dispatch();
    }
}
```

- `final readonly class` — same as `SepaExportServiceImpl`
- Injects `InvoiceBatchRepository` interface — not the Eloquent implementation

- Maps domain VOs (`InvoiceEmailRecipient`) to jobs — the mapping logic lives in infrastructure, not in the domain listener
- The `#[Autowire]` attribute on the interface auto-binds to this implementation via `jeroen-g/autowire`

## Key Technical Decisions

### 1. Custom Headers for Mailable Metadata
The `MessageSending` event only provides the Symfony `Message` object, not the original Mailable instance. A custom `X-Mailable-Class` header is added to the `Envelope` to bridge this gap. This is the cleanest approach that works with both sync and queued sending. Mailables without this header (e.g. Fortify notifications) are tracked with `mailable_class` as null.

### 2. Member Matching by Email
The `LogOutgoingMail` listener matches recipients to members by email address lookup. This is simple and effective. If a member changes their email, historical records retain the original email.

### 3. System Email Classification
Any email where `member_id` is null is classified as a "system" email. This includes admin notifications, Fortify password resets, and any other non-member mail. The `OutgoingEmailResource` provides tabs and filters to separate member vs. system emails.

### 4. Rate Limiting via Queue Middleware
The `RateLimited('invoice-emails')` middleware on `SendInvoiceEmail` ensures no more than 20 invoice emails per minute are sent. This prevents being flagged as spam. The rate limiter is defined in `AppServiceProvider::boot()` using `RateLimiter::for()`.

### 5. Job Batching via Domain Service
`Bus::batch()` creates a tracked batch in the `job_batches` table. This provides progress tracking, cancellation support, and visibility into which invoices have been emailed. The batch name `invoice-batch-{id}` makes it easy to identify. The `Bus::batch()` call is encapsulated within `InvoiceBatchEmailDispatcherImpl` (infrastructure layer), keeping the domain listener free of framework-specific batch dispatching logic. This follows the `SepaExportService`/`SepaExportServiceImpl` pattern.

### 6. DDD Service Layer for Batch Email Dispatch
The `QueueInvoiceEmails` listener delegates to `InvoiceBatchEmailDispatcher` (domain interface) → `InvoiceBatchEmailDispatcherImpl` (infrastructure implementation). The service retrieves invoice recipients via `InvoiceBatchRepository::getInvoiceEmailRecipients()` which returns `list<InvoiceEmailRecipient>` domain VOs — not Eloquent models. This keeps the listener thin (single responsibility: react to event) and the domain layer free of Eloquent and `Bus` facade dependencies.

### 7. Invoice Email Lines as Value Objects
Invoice line data passed to `InvoiceMail` uses `InvoiceEmailLine` VOs instead of associative arrays. Each VO holds typed domain data (`string $description`, `float $quantity`, `CompoundPrice $price`, `CompoundPrice $subtotal`) and provides a `formattedQuantity()` method. The Blade template accesses properties via `$line->description` and relies on `CompoundPrice::__toString()` for price formatting. This provides type safety, IDE autocompletion, and consistency with the existing VO patterns (`SepaExportInvoice`, `CompoundPrice`).

### 8. Event Discovery
Mail tracking listeners (`LogOutgoingMail`, `MarkMailAsSent`) are registered explicitly in `AppServiceProvider::boot()` because they listen to framework events (`MessageSending`, `MessageSent`), not domain events. The `QueueInvoiceEmails` listener is auto-discovered because it lives under `app/Domain/` and listens to a domain event (`InvoiceBatchClosed`).
